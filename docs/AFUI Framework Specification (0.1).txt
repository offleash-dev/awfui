AFUI Framework Specification (Draft 1.0)

1. Purpose
AFUI is a lightweight, RTOS‑friendly UI framework built on top of Adafruit_GFX.
It provides:
-A consistent widget model
-A screen/state system
-Dialogs and modal overlays
-A unified event model
-Optional double‑buffering via a single full‑screen canvas

The framework avoids heavy dependencies and is suitable for STM32, RP2040, ESP32, and similar MCUs.


2. Architecture Overview
Code
AFWorld
 └── AFScreenList
       └── AFScreen (active)
             ├── Widgets
             ├── Dialogs
             ├── Modal Dialog (optional)
             └── Fullscreen Dialog (optional)


3. Core Components
3.1 AFWorld
Top‑level runtime and orchestrator.

Responsibilities
Owns global theme
Owns AFScreenList
Runs the event loop
Polls hardware (touch, buttons, timers)
Converts hardware input → AFEvent
Dispatches events to active screen
Triggers redraws

Key API
v`oid setTheme(const AFTheme&)
void addScreen(AFScreen*)
void setActiveScreen(AFScreen*)
void loop()  // called from main or RTOS task


3.2 AFScreenList
Manages application screens.
Responsibilities
Stores screens
Tracks active screen
Handles screen switching
Optional transitions (fade, slide, etc.)

Key API
void add(AFScreen*)
void remove(AFScreen*)
void setActive(AFScreen*)
AFScreen* getActive() const


3.3 AFScreen
Represents a top‑level UI state.
Responsibilities
Owns widgets
Owns dialogs
Owns optional full‑screen canvas
Handles hit‑testing
Handles event dispatch
Handles modal blocking
Draws itself and children

Properties
Adafruit_GFX& display
GFXcanvas16* canvas (nullable)
std::vector<AFWidget*> widgets
std::vector<AFDialog*> dialogs
AFModalDialog* activeModal (nullable)

Key API
void addWidget(AFWidget*)
void addDialog(AFDialog*)
void showModal(AFModalDialog*)
void dismissModal(AFModalDialog*)
void handleEvent(const AFEvent&)
void draw()



4. Widget System
4.1 AFWidget (base class)
Geometry
int16_t x, y (upper‑left origin)
int16_t width, height
bool visible
AFWidget* parent
Responsibilities
Draw itself
Hit‑test
Receive events
Optional: request redraw

Virtual API
virtual void draw(Adafruit_GFX&) = 0
virtual bool hitTest(int16_t px, int16_t py)
virtual void onPress(const AFEvent&)
virtual void onRelease(const AFEvent&)
virtual void onClick(const AFEvent&)
virtual void onKey(const AFEvent&)
virtual void onButton(const AFEvent&)


4.2 AFDialog (inherits AFWidget)
Responsibilities
Container widget
Owns child widgets
Draws background/border
Routes events to children

API
void addChild(AFWidget*)
void draw(Adafruit_GFX&) override
void handleEvent(const AFEvent&)


4.3 AFModalDialog (inherits AFDialog)
Responsibilities
Captures all input
Blocks underlying widgets
Dismissed explicitly

API
void show(AFScreen&)
void dismiss()
bool isModal() const


4.4 AFFullscreenDialog (inherits AFModalDialog)
Purpose
A modal overlay that fills the entire screen but does not own its own canvas.

Behavior
x = 0, y = 0, width = screenWidth, height = screenHeight
No border/chrome by default
Drawn into the screen’s canvas
Blocks input while active
Ideal for menus, wizards, alerts, etc.


5. Event Model
5.1 AFEvent
Represents a hardware or logical event.

Fields
enum class Type { TouchDown, TouchUp, TouchMove, Button, Key, Timer }
Type type
int16_t x, y (touch)
uint8_t buttonId (hardware button)
uint16_t keycode (future keyboard)
uint32_t timestamp


5.2 Event Flow
AFWorld polls hardware
Builds AFEvent
Sends to active screen

AFScreen:
If modal dialog active → send to modal
Else → hit‑test dialogs
Else → hit‑test widgets

AFWidget receives event via virtual handlers


6. Rendering Model
6.1 AFScreen::draw()
Two modes:

Canvas mode
Clear canvas
Draw widgets/dialogs into canvas
Flush canvas to display

Direct mode
Draw widgets/dialogs directly to display


6.2 Redraw strategy
Initial version:
Full redraw each frame
Optional: per‑widget needsRedraw flag later
Optional: dirty‑rect tracking later


7. RTOS Integration
AFWorld::loop() runs in a dedicated UI task
Hardware input may be:
Polled in UI task
OR delivered via queue from ISR
No blocking calls inside widgets
Screen transitions and dialog operations are synchronous within UI task


8. Theme System
AFTheme
Colors
Fonts
Padding
Corner radius
Default widget styles

Stored in AFWorld and inherited by screens/widgets unless overridden.


9. Memory Model
Only one full‑screen canvas is active at a time (owned by AFScreen)
So only one double buffer normally.  If screen transitions are supported, a 
2nd buffer might temporarily exist until the transition is complete.

Dialogs and widgets do not allocate canvases
AFFullscreenDialog reuses the screen’s canvas
Screens are long‑lived; dialogs/widgets may be dynamic

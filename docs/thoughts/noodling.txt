// ------------------------------------------------------------
// AFDialogList - a future behavior to support layered UI
// ------------------------------------------------------------

template <size_t MAX_DIALOGS>
class AFDialogList {
public:
    using Container = AFVector<AFDialog*, MAX_DIALOGS>;

    bool add(AFDialog* d) {
        if (dialogs.full()) return false;
        dialogs.push_back(d);
        return true;
    }

    bool remove(AFDialog* d) {
        for (size_t i = 0; i < dialogs.size(); ++i) {
            if (dialogs[i] == d) {
                dialogs.erase(dialogs.begin() + i);
                return true;
            }
        }
        return false;
    }s

    // Topmost dialog hit-test
    AFDialog* hitTest(int16_t x, int16_t y) const {
        for (int i = dialogs.size() - 1; i >= 0; --i) {
            AFDialog* d = dialogs[i];
            if (d->isVisible() && d->hitTest(x, y)) {
                return d;
            }
        }
        return nullptr;
    }

    // Future: dialog stacking, animations, transitions
    void bringToFront(AFDialog* d) {
        remove(d);
        add(d);
    }

private:
    Container dialogs;
};



Why a dialog list becomes essential with top/bottom bars
Imagine a screen layout like:

Code
+--------------------------------------+
| [Top Bar: tabs, title, icons]        |  ← persistent UI layer
+--------------------------------------+
|                                      |
|      Main content / game / app       |
|                                      |
+--------------------------------------+
| [Bottom Bar: nav, status, buttons]   |  ← persistent UI layer
+--------------------------------------+
In AwFui terms:
The top bar is a dialog or panel
The bottom bar is a dialog or panel
The main content is either:
widgets, or
a custom drawing screen (game, canvas, etc.)

These bars are not modal.
They are not part of the widget tree.
They are not normal children of the main content.

So maybe this means a need for a dialog list. 

1. They draw above widgets
Top bar and bottom bar must always be on top.
Dialogs already draw after screen widgets.

2. They receive events before widgets
Touch on the top bar should not hit the game canvas.
Dialogs already get priority in event routing.

3. They can contain their own widgets
Buttons, labels, icons — all inside the bar.

4. They are independent of the main content
You can switch screens and keep the bars consistent.

5. They can be themed separately
Bars often have different colors, fonts, padding.

6. They could be animated.  Uh, really, but nice to have
Slide in/out, fade, resize — 


When you have multiple persistent dialogs, you need:

1. Z‑ordering for drawing and events
Top bar above bottom bar?
Bottom bar above floating toolbar?
DialogList can manage:
bringToFront
sendToBack
layering rules

DialogList can handle:
pointer capture
focus
keyboard routing (future)

DialogList can track:
persistent bars
temporary overlays
modal dialogs (separate path or order list with top z order at front of list)

2. Hit‑testing priority
Does the top bar get touches before bottom bar?  And eventually one has focus, so the panel has focus.
DialogList can encapsulate:
dialogList.hitTest(x, y)

3. Layout helpers
dockTop(dialog)
dockBottom(dialog)
dockLeft(dialog)
dockRight(dialog)

Does this belong in DialogList or AFScreen (as a helper? suggesting coorindates).

4. When we have dialogs/panels
AFScreen becomes cleaner:
dialogList.draw(gfx);
dialogList.handleEvent(e);
Instead of manually iterating.

AwFui is already structured to grow into this naturally — you’re just seeing the next layer of the architecture before you need it.

    // Focus management

    // Z-order

    // Event routing
    void handleEvent(const AFEvent& e);

    AFWidget* focused = nullptr;

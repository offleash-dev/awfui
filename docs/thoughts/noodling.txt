// ------------------------------------------------------------
// AFDialogList - a future behavior to support layered UI
// ------------------------------------------------------------

template <size_t MAX_DIALOGS>
class AFDialogList {
public:
    using Container = etl::vector<AFDialog*, MAX_DIALOGS>;

    bool add(AFDialog* d) {
        if (dialogs.full()) return false;
        dialogs.push_back(d);
        return true;
    }

    bool remove(AFDialog* d) {
        for (size_t i = 0; i < dialogs.size(); ++i) {
            if (dialogs[i] == d) {
                dialogs.erase(dialogs.begin() + i);
                return true;
            }
        }
        return false;
    }

    // Topmost dialog hit-test
    AFDialog* hitTest(int16_t x, int16_t y) const {
        for (int i = dialogs.size() - 1; i >= 0; --i) {
            AFDialog* d = dialogs[i];
            if (d->isVisible() && d->hitTest(x, y)) {
                return d;
            }
        }
        return nullptr;
    }

    // Future: dialog stacking, animations, transitions
    void bringToFront(AFDialog* d) {
        remove(d);
        add(d);
    }

private:
    Container dialogs;
};



hy a dialog list becomes essential with top/bottom bars
Imagine a screen layout like:

Code
+--------------------------------------+
| [Top Bar: tabs, title, icons]        |  ← persistent UI layer
+--------------------------------------+
|                                      |
|      Main content / game / app       |
|                                      |
+--------------------------------------+
| [Bottom Bar: nav, status, buttons]   |  ← persistent UI layer
+--------------------------------------+
In AwFui terms:
The top bar is a dialog
The bottom bar is a dialog
The main content is either:
widgets, or
a custom drawing screen (game, canvas, etc.)

These bars are not modal.
They are not part of the widget tree.
They are not children of the main content.

They are UI layers.

This implies need for a dialog list.

1. They draw above widgets
Top bar and bottom bar must always be on top.
Dialogs already draw after widgets.

2. They receive events before widgets
Touch on the top bar should not hit the game canvas.
Dialogs already get priority in event routing.

3. They can contain their own widgets
Buttons, labels, icons — all inside the bar.

4. They are independent of the main content
You can switch screens and keep the bars consistent.

5. They can be themed separately
Bars often have different colors, fonts, padding.

6. They can be animated
Slide in/out, fade, resize — dialogs support this naturally.


When you have multiple persistent dialogs, you suddenly need:

1. Z‑ordering
Top bar above bottom bar?
Bottom bar above floating toolbar?
DialogList can manage:
bringToFront
sendToBack
layering rules

2. Hit‑testing priority
Top bar should intercept touches before bottom bar.
DialogList can encapsulate:
dialogList.hitTest(x, y)

3. Layout helpers
You might want:
dockTop(dialog)
dockBottom(dialog)
dockLeft(dialog)
dockRight(dialog)

This belongs in DialogList, not AFScreen.

4. Persistent vs transient dialogs
DialogList can track:
persistent bars
temporary overlays
modal dialogs (separate path)

5. Event routing rules
DialogList can handle:
pointer capture
focus
keyboard routing (future)

Animation and transitions would probably need AFDialogList for
slide‑in top bar
fade‑in bottom bar
animated toolbars


Awfui doesn't need DialogList today, but it will when we add:
top bars
bottom bars
side panels
floating toolbars
HUD overlays

AFScreen becomes cleaner:
dialogList.draw(gfx);
dialogList.handleEvent(e);
Instead of manually iterating.

As soon as a screen has multiple persistent UI layers (like a top bar and bottom bar), a dedicated DialogList becomes the right abstraction.

It gives you:
layering
hit‑testing
focus
docking
animations
clean event routing

AwFui is already structured to grow into this naturally — you’re just seeing the next layer of the architecture before you need it.

// ------------------------------------------------------------
// AFWidgetList
//  future behavior to support AFTextEdit
// ------------------------------------------------------------
	

#pragma once

#include <etl/vector.h>
#include "AFWidget.h"

template <size_t MAX_WIDGETS>
class AFWidgetList {
public:
    bool add(AFWidget* w);
    bool remove(AFWidget* w);

    AFWidget* hitTest(int16_t x, int16_t y) const;

    // Focus management
    void setFocus(AFWidget* w);
    AFWidget* getFocus() const;

    // Pointer capture
    void capturePointer(AFWidget* w);
    void releasePointer(AFWidget* w);
    AFWidget* getPointerCapture() const;

    // Z-order
    void bringToFront(AFWidget* w);
    void sendToBack(AFWidget* w);

    // Event routing
    void handleEvent(const AFEvent& e);

private:
    etl::vector<AFWidget*, MAX_WIDGETS> widgets;
    AFWidget* focused = nullptr;
    AFWidget* captured = nullptr;
};

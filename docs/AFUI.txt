
AwFui - A modern wrapper around Adafruit_GFX

1. Goals
Lightweight UI framework on top of Adafruit_GFX

RTOS‑friendly: explicit event loop, no hidden blocking

Consistent coordinates: all widgets use upper‑left origin

Single‑canvas model: at most one full‑screen canvas per screen

Clear separation of:

Application state (AFScreen)

Overlays (AFDialog, AFModalDialog, AFFullscreenDialog)

Elements (AFWidget and descendants)



2. Core concepts
AFWorld: top‑level runtime and event loop

AFScreenList: manages screens and active screen selection

AFScreen: root of a widget tree for a given app state

AFWidget: base class for all UI elements

AFDialog / AFModalDialog / AFFullscreenDialog: container widgets with different behavior

Theme: shared colors, fonts, spacing, etc.

Canvas: optional off‑screen framebuffer owned by AFScreen



3. Class specs
3.1 AFWorld
Responsibilities:

Owns:

AFScreenList

Global AFTheme

References to hardware interfaces (display, touch, buttons, etc.)

Runs the main UI loop:

Poll input

Build AFEvents

Dispatch to active AFScreen

Trigger redraws as needed

Key API:

void setTheme(const AFTheme& theme);

bool addScreen(AFScreen* screen);

void setActiveScreen(AFScreen* screen);

void loop();  // called from main or RTOS task


3.2 AFScreenList
Responsibilities:

Owns zero or more AFScreen instances

Tracks the active screen

Provides screen transitions (optional)

Key API:

bool add(AFScreen* screen);

void remove(AFScreen* screen);

void setActive(AFScreen* screen);

AFScreen* getActive() const;


3.3 AFScreen
Responsibilities:

Represents a top‑level UI state

Owns:

Optional GFXcanvas* (for double buffering)

Root‑level widgets

Dialogs (modal and non‑modal)

Handles:

Hit‑testing

Event dispatch

Drawing

Modal blocking

Key properties:

Adafruit_GFX& display;

GFXcanvas16* canvas; // nullable

std::vector<AFWidget*> widgets;

AFModalDialog* activeModal; // nullable

Key API:

void addWidget(AFWidget* w);

void addDialog(AFDialog* d);

void showModal(AFModalDialog* d);

void dismissModal(AFModalDialog* d);

void handleEvent(const AFEvent& e);

void draw();  // draws to canvas or directly to display


3.4 AFWidget
Responsibilities:

Base class for all visible elements

Defines geometry, visibility, and event hooks.  Subclasses should provide event hook callbacks for the primary event to avoid inheritance.

Key properties:

int16_t x, y, width, height;  // upper‑left origin

bool visible;

AFWidget* parent;

Key API (virtual):

virtual void draw(Adafruit_GFX& gfx) = 0;

virtual bool hitTest(int16_t px, int16_t py) const;

virtual void onPress(const AFEvent& e);

virtual void onRelease(const AFEvent& e);

virtual void onClick(const AFEvent& e);

virtual void onKey(const AFEvent& e);   // future

virtual void onButton(const AFEvent& e); // hardware button


3.5 AFDialog
Responsibilities:

Container widget with children

Draws background/border

Routes events to children

Key properties:

std::vector<AFWidget*> children;

Key API:

void addChild(AFWidget* w);

void draw(Adafruit_GFX& gfx) override;

void handleEvent(const AFEvent& e);  // hit‑test children

3.6 AFModalDialog (inherits AFDialog)
Responsibilities:

Same as AFDialog, but:

Captures all input while active

Dismissed explicitly

Key API:

void show(AFScreen& screen);

void dismiss();

bool isModal() const { return true; }


3.7 AFFullscreenDialog (inherits AFModalDialog or AFDialog)
Responsibilities:

Fills entire screen

No border/chrome by default

Uses screen’s canvas/display (no own canvas)

Behaves like a modal overlay that covers everything

Key behavior:

x = 0, y = 0, width = screenWidth, height = screenHeight

Optional dimming of background

All events routed here while active (if modal)



4. Event model
4.1 AFEvent
Fields (example):

enum class Type { TouchDown, TouchUp, TouchMove, Button, Key, Timer };

Type type;

int16_t x, y;        // for touch

uint8_t buttonId;    // for hardware buttons

uint16_t keycode;    // for keyboard

uint32_t timestamp;


4.2 Flow
AFWorld polls hardware → builds AFEvent

AFWorld → activeScreen->handleEvent(e)

AFScreen:

If activeModal → send to modal

Else → hit‑test dialogs/widgets

AFWidget receives event via onPress/onRelease/onClick/...



5. Rendering model
AFScreen::draw():

If canvas:

Clear canvas

Draw widgets/dialogs into canvas

Flush canvas to display

Else:

Draw widgets/dialogs directly to display

Optional:

Dirty‑rect tracking later

Simple “needsRedraw” flag per widget/screen initially



6. RTOS considerations
AFWorld::loop() is called from a dedicated UI task

No blocking calls inside widgets

Hardware input can be:

Polled in UI task, or

Posted into a queue that UI task drains into AFEvents

Screen transitions and dialog show/dismiss are synchronous from the UI task’s perspective
